The Two-Step Flow

1

Agent registers on Base

Mints an ERC-721 identity NFT via register(agentURI). The URI points to a JSON file describing the agent's capabilities, services, and endpoints.

2

Reputation accumulates

Other agents and users submit feedback via giveFeedback(agentId, score). Scores are aggregated on-chain into an average reputation.

3

Bridge identity (one-time)

Agent calls bridgeIdentity(agentId, dstEid). The contract reads both registries, encodes the snapshot, and sends it via LayerZero. DVNs independently verify the message. ~30 seconds.

4

Verify forever (any contract)

Any contract on the destination chain can call verifyAgent(address) or isReputable(address, minScore) to check the agent's identity and reputation. No further action needed from the agent.

5

Re-bridge to refresh

If reputation changes on Base, the agent bridges again to update the cached data. Each bridge includes a bridgedAt timestamp so consumers can check freshness.

Data Freshness — The Consumer Decides

Every bridged identity includes a bridgedAt timestamp. Consuming contracts choose their own freshness requirements.

High-Stakes Vault
< 1 hour
Large deposits require very recent identity data. Agent re-bridges frequently.
Agent Marketplace
< 24 hours
Service discovery accepts day-old reputation. Good enough for most interactions.
Access Gate
Any age
Just needs to know the agent exists and has some reputation. Bridge once, access forever.
